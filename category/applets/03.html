# 背景

# 开发新项目遇到的问题

  • 开发时间短

  • 要求技术栈完整

  • 达到辅助功能丰富

  • 针对不同的项目需求和团队的情况，搭建项目基础环境不能一成不变

# 目前项目的痛点

  • 重复性工作过多

  • 不便团队统一开发风格，及跨团队合作

  • 不便插件扩展等功能

# 期望脚手架解决的问题

  1. 快速搭建企业级项目框架
  2. 减少重复性工作
  3. 统一团队开发风格，便于跨团队开发及后期维护，降低新人上手成本
  4. 提供一键前端项目的创建，配置，本地开发，插件扩展等功能，让开发者更多时间专注于业务
  5. 规范项目开发目录

# 脚手架基础架构

# 站在前端研发的视角，分析开发脚手架的必要性

A. 提升前端研发效能【提炼通用代码、通用流程、构建发布上线】

B. 脚手架核心价值：自动化、标准化、数据化

C. 和自动化构建工具(jenkins、travia)区别：自动化构建工具在服务端执行，无法覆盖本地操作且定制自动化的构建工具需要用到Java等后端语言，对前端不友好

# 从应用角度看CLI开发脚手架过程

  1. 首先是个npm项目，项目中有一个bin/vue.js的文件，且这个项目发布到了npm上

  2. 将npm项目安装到了lib/node_modules

  3. 在node的bin目录下配置软链接到lib/node_modules/@vue/cli/bin/vue.js
